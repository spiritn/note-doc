## 数据库三范式

### 第一范式

所有的字段都是基本字段，最小的信息单元，不可进一步拆分

### 第二范式

在满足第一范式的基础上，每一条记录都有主关键字段（可能由两个字段组合而成）。而且所有字段都必须完全依赖主键，不能部分依赖主键

不符合第二范式的例子

| 货物类型 | 货物ID | 货物名称   | 注意事项           |
| -------- | ------ | ---------- | ------------------ |
| 瓷碗     | 1      | 白色瓷碗   | 易碎品             |
| 瓷碗     | 2      | 青花瓷碗   | 易碎品             | 
| 瓷碗     | 3      | 雕花瓷碗   | 易碎品             |
| 三合板   | 4      | 普通三合板 | 易燃物品，注意防火 |

在该表中**主键为（货物类型，货物ID）**，货物名称字段完全依赖于这个主键，换句话说，货物的名称完全是取决于这个主键的值的。但**“注意事项”这一列，仅依赖于一个主键中”货物类型“这一个属性。**简单地说，第二范式要求每个非主属性完全依赖于主键，而不是仅依赖于其中一部分属性。

### 第三范式

在满足第二范式的基础上，不能包含那些依赖非主键字段的字段，或者说由非主键字段衍生出来的字段


但是我们有时会根据**业务优先**的原则，进行反范式的处理。比如违反第三范式，做一些冗余字段，比如一张表除了person_id字段，我们还冗余person_name字段，方便业务查询，提高查询效率。

## ER图

一般用于数据库的设计阶段，用来帮助梳理和理解表结构和关系。

包含实体，属性和关系。

强实体：不依赖于其他实体的实体。

弱实体：依赖于另一个实体，不能独立存在

关系包含 一对一，一对多，多对多。




InnoDB_buffer_pool_size 表示用于缓存的大小，一般数据库用的都是专用服务器，甚至128G。适当调大可以充分利用内存

InnoDB_buffer_pool_instances表示buffer pool划分为几个部分，

InnoDB_flush_log_at_trx_commit默认1表示每次提交事务的时候，都写入binlog写入磁盘。可以改成2，表示每次提交事务都写入binlog，但是每隔一秒写入磁盘。
